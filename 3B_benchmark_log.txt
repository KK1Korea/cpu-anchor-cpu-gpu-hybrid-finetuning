  ■ CRITICAL FINDING — train_loss -22.7% Is a Measurement Artifact:

    The -22.7% comes from how PyTorch trainer computes train_loss (average of ALL steps).

    Phase1 cosine completes in 100 steps:
      → lr drops to ~0 by step 70-100
      → loss at steps 70-100: 1.155~1.182 (low, but model stopped learning)
      → These artificially low Phase1 values pull the overall average down

    Baseline cosine spans 500 steps:
      → lr at steps 70-100 still 1.947~1.966e-05 (active learning)
      → loss at steps 70-100: 1.137~1.264 (similar range, but lr still high)
      → No artificially low "stalled" steps

    Step-by-step Phase2 truth: Exp A is consistently ABOVE Baseline (+0.001~+0.036).
    The transition PROVIDED NO BENEFIT — it caused shock and permanent (tiny) deficit.

    7B's -22.5% must be re-examined:
      → 7B Phase1 also used 100-step cosine → same averaging artifact
      → BUT 7B had QLoRA 4-bit → precision×quantization interaction may have
        created REAL Phase2 improvement that 3B doesn't have
      → 7B step-by-step data for Exp A (GPU-only) was not fully logged (JF-2),
        making direct Phase2 comparison impossible

  ■ FINDING — Transition Effect Without Quantization = No Real Improvement:

    3B (16-bit full, no quantization):
      bf16 = fp32 confirmed (BF-3)
      fp32→bf16 transition = no Phase2 improvement, only shock damage
      train_loss -22.7% is lr schedule artifact, not genuine improvement

    7B (QLoRA 4-bit):
      bf16 ≠ fp32 under quantization (inferred — not directly tested)
      fp32→bf16 transition = -22.5% (artifact + possible real QLoRA interaction)
      Needs 7B step-by-step A vs F Phase2 re-examination

    Emerging hypothesis:
      Precision-staged training benefit = f(quantization_level)
      More quantization → more benefit from fp32 staging
      No quantization → no benefit (possibly harmful due to lr shock)

    setup: 7B GPU used batch 1×8, 7B CPU used batch 2×4.

  ■ FINDING — Batch Confound Discovered:

    ┌─────────────────────┬───────────┬───────────┬──────────┐
    │ Experiment          │ GPU batch │ CPU batch │ CPU vs GPU│
    ├─────────────────────┼───────────┼───────────┼──────────┤
    │ 3B 16-bit (BF-6,7) │ 1×8       │ 1×8       │ = (0.002)│

    └─────────────────────┴───────────┴───────────┴──────────┘

    When batch is controlled (same for CPU and GPU):
      → CPU = GPU in ALL conditions: 16-bit, 4-bit, continuous lr, discontinuous lr

    When batch differs (1×8 vs 2×4):
      → CPU ≠ GPU (7B only case where batch differed)

    CONCLUSION: The 7B "0.82% CPU deeper" finding was likely caused by
    micro-batch mismatch (2×4 vs 1×8), NOT by CPU determinism.

    This CANNOT be verified on 7B because GPU fp32 with batch 2×4 would OOM
    on 24GB VRAM (7B × 4 bytes = 28GB + optimizer states).
