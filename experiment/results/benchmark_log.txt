[JF-1a] Key Design Decision: CPU Phase fp32 — 2026-03-01
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

  ■ Decision:
    Changed Experiment C Phase 1 (CPU anchor phase) precision from bf16 → fp32.

  ■ Additional Decision:
    Changed GPU phases (all of Experiment A + Experiment C Phase2) precision from fp16 → bf16.
    bf16 is the LLM fine-tuning industry standard (8 exponent bits = wider range, Qwen2.5 pre-train format).
    RTX 5090 has full bf16 support. fp16 has overflow risk, unsuitable for LLM training.

  ■ Changes:
    jellyfish_exp_c_phase1.yaml:
      Before: bf16: true
      After:  bf16: false / fp16: false  (= fp32)

    jellyfish_exp_a.yaml:
      Before: fp16: true
      After:  bf16: true

    jellyfish_exp_c_phase2.yaml:
      Before: fp16: true
      After:  bf16: true

  ■ This Is the Core of the Experiment:
    All hyperparameters between A (GPU) and C (CPU) are completely identical.
    lora_rank, lora_alpha, lora_dropout, batch_size, gradient_accumulation,
    learning_rate, lr_scheduler, warmup_steps, seed, quantization — all the same.

    The only difference:
      Experiment A (GPU): bf16  →  Industry-standard half-precision + GPU hardware noise
      Experiment C Phase1 (CPU): fp32  →  Maximum precision + deterministic computation

    The hypothesis itself is "CPU deterministic precision anchors the model",
    so running CPU in bf16 (half-precision) would undermine CPU's precision advantage.
    fp32 ensures the maximum-precision gradient for anchor establishment.

    Integer and decimal analogy:
      fp32 CPU = 1, 2, 3 ... striking exact integers
      bf16 GPU = 1.1, 1.2, 1.01 ... filling between integers
      bf16 CPU = 0.97, 2.03 ... ← can't even hit integers. Pointless.

  ■ Resource Impact:
    7B QLoRA fp32 computation: model itself loaded in 4-bit (~5GB)
    LoRA adapter + optimizer in fp32 → increased RAM usage
    Sufficient headroom in 64GB RAM. Only 100 steps, so time is manageable.

    │                  │ Experiment A │ Experiment C                 │

  ■ Key Observations:

    1. CPU fp32 100 steps > GPU bf16 500 steps
       - CPU Phase1 reached A's final value (1.184) at step 50
       - CPU Phase1 reached 1.090 at step 80 = below A's final value
       - 5x fewer steps, lower loss = deterministic fp32 gradient efficiency

    2. Zero Transition Shock
       - Phase1 final 1.192 → Phase2 first 1.209 (difference +0.017)
       - GPU respected CPU anchor. Optimizer state continuity maintained.

    3. Anchor Basin Held
       - Throughout Phase2, GPU stayed within CPU anchor range (~1.08-1.27)
       - Spikes (e.g. 1.295) immediately returned to anchor range
       - CPU-established basin walls contained GPU noise

    4. GPU Hardware Noise Floor Demonstrated
       - lr decreased 5x but loss oscillation only decreased 30%
       - At lr → 0 (1.117e-09), loss still oscillated ~0.17
       - Residual noise not removable by software (lr decay) = hardware origin

    5. Deep Exploration Within Anchor
       - GPU found deeper points than CPU's lowest (1.090) → 0.9966
       - CPU determined the basin → GPU explored deeper within it
       - Exactly the "strike integers, then fill decimals" structure from the hypothesis

  ■ Interim Conclusion:
    CPU anchor validity confirmed by train loss.
    22.5% loss reduction is a dramatic difference in a 500-step mini-experiment.

    ⚠ However, this alone is not conclusive:
      - train loss ≠ model quality (overfitting possible)
      - Benchmark (MMLU etc.) not yet verified
      - No repeated experiments (seed 42 only)
      - 500-step mini scale

  ■ Background:

    3B experiments (BF-8) discovered that CPU = GPU in QLoRA 4-bit when batch
    is controlled (1×8 for both). This suggested the 7B "0.82% CPU deeper"
    finding was a batch artifact (F: 1×8 vs C/G: 2×4).

    The question was: can this be confirmed on 7B directly?
    Answer: YES — Exp A (GPU bf16, batch 2×4) provides the missing comparison.

  ■ Key Insight:

    A (GPU bf16, batch 2×4) shares batch configuration with C/G (CPU fp32, batch 2×4).
    A differs from C/G in: device (GPU vs CPU) AND precision (bf16 vs fp32).
    Since bf16 = fp32 (confirmed 3× on 3B + now on 7B too), the only real
    variable between A and C/G is device (CPU vs GPU).

    If A ≈ C/G → CPU = GPU on 7B too (same batch → same result).
    If A ≉ C/G → something else is different (unexpected).
